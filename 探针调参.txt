## 这次代码下，哪些判断要修正

第一，**当前模型不是“没有爬升”**。
我直接重看了你上传的 `eval_metrics.csv`。如果只看 full-eval checkpoint，模型从 `epoch 0` 到中后期其实有明显提升：
`epoch 0` 大约是 `merge=0.15 / endpoint=0.45 / safety=0.20 / collision=0.30 / eval_ep_rew=0.50 / dense_norm100=32.8`；
到 `epoch 200` 已经到 `0.75 / 1.00 / 0.65 / 0.35 / 1.525 / 36.0`；
`epoch 240` 是这轮最像 best checkpoint 的点，约 `0.75 / 1.00 / 0.90 / 0.10 / 1.775 / 40.6`；
`epoch 300` 则是 `0.60 / 0.90 / 0.85 / 0.10 / 1.60 / 40.4`。
所以这轮更像是“**先明显提升，后面轻微回落**”，不是“从头到尾都没学到”。

第二，**`Policy_Return_Components_Norm100` 这张图本来就不该长成经典 RL 那种平滑爬升曲线**。
因为它本质上就是
[
25\cdot merge_success + 25\cdot endpoint_success + 50\cdot safety_success
]
是三条 success rate 的离散加权和，而不是 dense return。当前 `evaluate_policy_metrics()` 里，`eval_ep_rew_mean` 也是直接把环境 `step()` 返回的 `reward` 累加起来；而环境 `step()` 返回的正是那个稀疏 `r_goal`。所以这张 component 图天然会是**台阶化、抖动、可回落**的，不能拿它当“主学习曲线”。([GitHub][3])

第三，**现在 raw dense score 仍然不会从 0 开始**。
原因不是代码错，而是设计上如此：`reset()` 还是从真实专家轨迹的起始位置和速度起跑，不是随机乱状态；而 dense reward 里 `p_goal` 和 `p_lane` 只要车还在向正确方向推进，就会持续给正分。也就是说，哪怕是未训练策略，也会有一个不低的 eval-only dense baseline。你如果想让主图从 0 起步，仍然要画 **random-normalized score**，而不是原始 `eval_dense_return_norm100`。([GitHub][1])

第四，**`paper_rank_score_mean` 也不适合作为主 learning curve**。
它对应的是论文里的 trajectory ranking score，
[
(\bar v_y-\sigma_{v_y})+\overline{TTC}-5\overline{|jerk|}
]
本质上是整条轨迹的效率/安全/舒适 summary，不是训练 return。论文主评估也不是“reward 曲线是否单调上升”，而是 merging success、平均速度、速度波动、TTC、headway、collision、jerk 这些指标。你现在代码里 `paper_rank_score_mean` 更适合当辅助指标，而不是主图。

## 现在要不要“大调参”

我的建议是：

**要做轻量 probe，但先不要大改模型结构。**

原因很简单。
当前 repo 的主结构已经比较完整了：

* 注意力开着
* goal condition 开着
* safety module / safety branch / safety aux 都开着
* `goal_bonus = 0.5`
* `n_disc_updates_per_round = 6`
* `PPO_EPOCHS = 8`
* `GENERATOR_LEARNING_RATE = 8e-5`
* `DISCRIMINATOR_LEARNING_RATE = 5e-5`
* reward normalization 仍是关的。([GitHub][4]) 

而且 safety 这条线当前是**整轮 frozen**。配置里 `SAFETY_UNFREEZE_TIMESTEPS = EPOCHS * STEPS_PER_EPOCH + 1`，再配合你当前 `EPOCHS=300, STEPS_PER_EPOCH=2048`，意味着 unfreeze 条件根本不会触发，所以整个 run 都停在 frozen safety prior。换句话说，这轮的主问题并不是“安全分支后期乱跳”，而更像是 **PPO 更新节奏 + 对抗平衡中期失稳**。([GitHub][4]) 

所以当前最值得扫的，不是 architecture，不是 goal_bonus，也不是 safety 结构，而是**训练节奏参数**。

## 我建议的 probe 顺序

先固定不动：

* `ENABLE_ATTENTION = True`
* `ENABLE_GOAL_CONDITION = True`
* `ENABLE_SAFETY_MODULE = True`
* `ENABLE_SAFETY_BRANCH = True`
* `ENABLE_SAFETY_AUX_LOSS = True`
* `goal_bonus = 0.5`
* `n_disc_updates_per_round = 6`
* safety phase 继续 frozen

先扫这三类最可能改变“中期塌一下、后面才恢复”现象的参数：

### 第一优先级

`PPO_EPOCHS: 8 -> 6`

这是我最先试的。你当前 repo 里 PPO 每轮 epoch 数偏高，而你这轮最明显的问题恰恰出现在 critic 追 moving reward、actor 又几乎不动的区间。先把每轮 PPO 迭代减一点，通常最容易让曲线更稳。([GitHub][4])

### 第二优先级

`GENERATOR_LEARNING_RATE: 8e-5 -> 7e-5`

这会比直接大动判别器更稳。你现在后面其实能恢复，说明 generator 不是完全学不动，而是节奏偏冲。

### 第三优先级

只开一次
`ENABLE_REWARD_NORMALIZATION = True`

因为你当前 repo 已经接了 `NormalizedRewardNet`，这是最温和的“稳定 reward 尺度”手段。它比去改 raw reward 符号、镜像曲线都干净得多。([GitHub][3])

我建议 probe 就跑这 4 组：

* A: `PPO_EPOCHS=8, GENERATOR_LR=8e-5`，当前 baseline
* B: `PPO_EPOCHS=6, GENERATOR_LR=8e-5`
* C: `PPO_EPOCHS=6, GENERATOR_LR=7e-5`
* D: `PPO_EPOCHS=8, GENERATOR_LR=7e-5`

如果 A/B/C/D 都还是中期明显塌，再单独补一个：

* E: 在 C 基础上 `ENABLE_REWARD_NORMALIZATION=True`

不太方便的地方是：

最省事的 probe 做法是：

把 `train_airl_baseline.py` 里这几行临时改成：

```python
save_freq_epochs = 10
eval_freq_epochs = 1
quick_eval_episodes = 4
full_eval_episodes = 10
```

再把 `Config` 里的：

```python
EPOCHS = 100
```

这样你一次 probe 只跑 100 epoch，但会有：

* epoch 0 一次 full eval
* 每个 epoch 一次 quick eval
* 每 10 epoch 一次 full eval + save

这对看“中期会不会塌”和“后段能不能回稳”已经够用了。
