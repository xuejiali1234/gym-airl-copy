# Reproducibility Guide

## 这份文档是干什么的

这份文档专门解释一个很常见、也很容易让人困惑的问题：

- 为什么我明明在 `config.py` 里写了 `SEED = 42`，实验还是复现不了？
- 为什么同样的参数，昨天能跑出很好的结果，今天再跑一遍却差很多？
- 强化学习是不是本来就“看运气”，所以没法复现？

短答案是：

- 之前这套代码里，`SEED = 42` 只固定了很少一部分流程。
- 真正训练时还有多处随机源没有被固定。
- 对 AIRL 这种“策略网络 + 判别器 + 环境采样”同时在动的对抗训练来说，这些随机性会被明显放大。

所以之前的问题不是“你记错了参数”，而是：

- **训练本体并没有被真正固定住。**


## 先说结论

之前实验难以复现，主要不是因为数据集顺序乱了，而是因为下面这些随机源没有统一控制：

1. 网络初始化随机
2. PPO 内部采样随机
3. 环境每回合抽哪条轨迹随机
4. 安全模块预训练时的负样本合成随机
5. `DataLoader(shuffle=True)` 的打乱顺序随机

也就是说，过去虽然写了一个 `SEED = 42`，但它**没有真正覆盖整条训练链路**。


## 为什么“固定了训练种子”还是无法复现

这是最关键的误区。

很多人会以为：

```python
SEED = 42
```

就等于“整个实验固定了”。

实际上不是。

“设置一个 seed”只是在说：

- 如果某个模块愿意使用这颗 seed，
- 它的随机行为就能被固定。

问题在于：

- **不是所有模块都会自动去读这颗 seed。**

在之前的代码里，`SEED` 主要只真正用于两件事：

1. 划分训练集 / 验证集
2. 周期评估时 `eval_env.reset(seed=...)`

这只能保证：

- 训练/验证数据拆分比较稳定
- 周期评估本身比较稳定

但**不能保证训练过程稳定**。

而训练过程恰恰是最重要、也最随机的部分。


## 之前到底有哪些地方在“偷偷随机”

### 1. 训练入口没有统一播种

之前没有在训练开始时统一做下面这些事：

- `random.seed(SEED)`
- `np.random.seed(SEED)`
- `torch.manual_seed(SEED)`

这会导致：

- Python 自带随机数不固定
- NumPy 随机数不固定
- PyTorch 权重初始化和很多采样过程不固定

直接后果是：

- 同一个模型结构，每次初始权重都可能不同
- 同一组参数，会从不同起点开始训练


### 2. PPO 自己没有显式拿到 seed

即使外面设置了一些随机种子，强化学习框架内部也不一定自动继承。

之前 `PPO(...)` 没有显式传：

```python
seed=cfg.SEED
```

这意味着：

- 策略初始化
- rollout 相关采样
- 框架内部一些随机行为

仍然可能漂。

所以以前是：

- 表面上写了 `SEED = 42`
- 但 PPO 这层并没有被真正固定


### 3. 环境 reset 时用的是全局 `np.random`

之前环境在 `reset()` 里会这样做：

- 修改全局 `np.random.seed(seed)`
- 用 `np.random.randint(...)` 随机挑一条轨迹

这有两个问题：

1. 全局 NumPy 随机状态很容易被别的地方污染
2. 谁先抽一次、后抽一次，都会影响后面的结果

这会导致：

- 训练时 episode 顺序不稳定
- 即使参数一样，也可能看到完全不同的轨迹采样路径

对强化学习来说，这非常致命，因为：

- 训练早期看到什么样的样本
- 会强烈影响后面学成什么策略


### 4. 安全模块预训练本身也是随机的

之前安全模块预训练里有多处随机操作：

- 从安全状态里随机挑样本
- 随机生成更激进的候选动作
- `torch.randperm(...)` 下采样
- `DataLoader(shuffle=True)` 打乱顺序

但这些都没有绑定统一的 seed。

这意味着：

- 即使主训练参数完全一样
- 每次训练前预训练出来的安全网络也可能不是同一个

换句话说：

- 你不是在“同一个起点”上继续训练
- 而是在“每次先造一个不一样的安全老师”，再开始主训练


### 5. AIRL 本身对随机扰动更敏感

这点特别重要。

普通监督学习里，随机性有时候只会造成一点小波动。

但你这里是 AIRL / 对抗训练，里面同时有：

- 生成器（PPO）
- 判别器（reward/discriminator）
- 环境采样
- 安全模块先验

这些模块互相影响，像一个连锁系统。

所以哪怕只是：

- 初始权重稍微不同
- 前几轮环境抽样顺序不同
- 安全预训练样本稍微不一样

最后都可能被放大成：

- 一次训练很好
- 下一次训练很差

这也是为什么你之前会感觉：

- “同一组参数，昨天很好，今天又完全不行”


## 之前什么是稳定的，什么不是

### 相对稳定的部分

这些之前其实已经比较稳定：

1. 数据文件读取顺序
   - `data_loader.py` 里对文件列表做了排序

2. 训练集 / 验证集划分
   - `split_dataset(..., seed=cfg.SEED)` 是固定的

3. 周期评估
   - 每个评估 episode 都显式使用了 `seed`

### 不稳定的部分

这些才是复现失败的主要来源：

1. 主训练入口没有统一播种
2. PPO 没有显式 seed
3. 环境训练采样依赖全局 `np.random`
4. 安全预训练采样和打乱顺序没固定


## 这次具体做了哪些修复

### 修复 1：训练入口统一播种

现在在训练开始时会统一设置：

- Python 随机数
- NumPy 随机数
- PyTorch 随机数

同时还会尽量打开 PyTorch 的确定性模式。

这一步的作用是：

- 固定整轮训练的随机起点


### 修复 2：PPO 显式接收 seed

现在 `PPO(...)` 会显式拿到：

```python
seed=cfg.SEED
```

这一步的作用是：

- 固定 SB3/PPO 框架内部那层随机性


### 修复 3：环境改用自己的 RNG

现在环境不再依赖全局 `np.random`，而是用 Gymnasium 环境自己的 RNG。

这一步的作用是：

- 训练 episode 采样更稳定
- 不再受全局 NumPy 状态污染


### 修复 4：安全预训练的随机采样绑 seed

现在安全预训练里：

- 合成不安全动作
- 随机下采样
- `DataLoader(shuffle=True)`

都绑定到了同一颗 seed。

这一步的作用是：

- 安全网络预训练更接近可复现
- 主训练前的 safety prior 更稳定


### 修复 5：每轮日志保存完整参数快照

现在每轮日志目录下都会自动写一个：

- `run_config.json`

里面会保存：

- 当前 config
- 当前真正生效的关键参数
- 参考目标实验信息

这一步不是“修随机”，而是“修记录”。

它解决的问题是：

- 以后不用再靠回忆猜“那轮到底是 5e-5 还是 7e-5”
- 日志和参数能一一对应


## 修复后是不是就 100% 完全复现了

不能这么承诺。

更准确地说：

- **现在比之前“可复现得多”，而且实验之间终于可比较了。**

但强化学习里，即使做了这些修复，仍然可能存在少量残余差异，来源包括：

1. 不同硬件
2. 不同 CUDA / PyTorch 版本
3. 某些底层算子的非完全确定性
4. 对抗训练本身的高敏感性

所以正确预期应该是：

- 修复前：同参数结果可能天差地别，几乎没法调参
- 修复后：同参数结果应该明显更稳定，至少能判断“这个旋钮到底有用没用”


## 一个常见误解：强化学习是不是应该先随机找好的，再固定

这个说法有一半对，一半不对。

### 对的部分

在探索阶段，确实可以接受一些随机性。

因为强化学习本来就会出现：

- 某个 seed 特别幸运
- 某个 seed 特别倒霉

有时候跑多个 seed，能帮助我们看到某组参数“有没有潜力”。

### 不对的部分

如果你在调参阶段还不固定随机性，就会出现：

- 今天觉得 `5e-5` 好
- 明天觉得 `7e-5` 好
- 后天又觉得 `5e-5` 更差

最后你根本不知道：

- 是参数真的更好
- 还是只是随机更走运

所以正确流程应该是：

### 阶段 1：固定一个 seed 做单旋钮调参

例如一直用：

- `seed = 42`

目标是：

- 先把大方向调顺
- 让每次实验真正可比较

### 阶段 2：挑出候选最优参数，再做多 seed 复验

例如再跑：

- `42`
- `52`
- `62`

目标是：

- 看这组参数是不是只对某一个 seed 好
- 还是整体都比较稳

所以：

- **固定 seed 不是让模型“找不到好结果”**
- 而是让你知道“到底是不是这组参数真的更好”**


## 为什么之前最好的结果不能靠 checkpoint zip 完整恢复

这个问题也容易误解。

你可能会觉得：

- 既然保存了很多 `.zip`
- 那是不是能直接回到当时训练最好的完整状态？

不完全行。

原因是这些 `.zip` 主要保存的是：

- SB3 policy
- 优化器状态
- 一些策略侧信息

但它**不能完整保存 AIRL 外层所有训练态**，例如：

- 判别器当前状态
- reward net 外层包装逻辑
- 对抗训练过程中的一些外部配置
- 当时完整的训练随机轨迹

所以 checkpoint 可以帮你：

- 恢复一个当时的策略模型

但不能保证：

- 从这个 zip 出发，继续训练就一定走回原来那条轨迹


## 现在怎么做实验才是对的

建议按下面这个流程。

### 做参数探索时

1. 固定 `SEED = 42`
2. 一次只改一个旋钮
3. 每轮保留 `run_config.json`
4. 看 `eval_metrics.csv` 和 `progress.csv`

这样你才能判断：

- 这次变好，到底是参数有效
- 还是运气更好

### 选出候选最优参数后

再做多 seed 实验，例如：

- `SEED = 42`
- `SEED = 52`
- `SEED = 62`

如果三次都不错，才能比较有信心地说：

- 这组参数不只是“撞中了一个好运种子”


## 当前这次修复后，你应该期待什么

修复后你不该期待的是：

- “以后每次都一模一样到小数点后全部一致”

你应该期待的是：

1. 同一组参数的波动明显变小
2. 一次只动一个参数时，趋势更可信
3. 好结果更容易复现到“同一档位”
4. 再也不用靠记忆猜某轮到底用了什么参数


## 一句话总结

之前实验无法复现，不是因为你没有写 `SEED = 42`，而是因为：

- **这颗 seed 没有真正贯穿整个训练链路。**

现在的修复思路就是：

- 把训练入口、PPO、环境采样、安全预训练、日志记录全部串到同一条可控链路上

这样后续我们才能真正回答一个问题：

- 这组参数，到底是真的更好，还是只是刚好更幸运。
