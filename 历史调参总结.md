## 3. 指标解释与选择规则

主要看这些指标：

| 指标 | 含义 | 选择倾向 |
|---|---|---|
| eval_ep_rew_mean | 原始环境 policy return，主要成功回报 | 越高越好 |
| eval_dense_return_norm100 | eval-only dense reward 归一化分数 | 越高越好 |
| paper_rank_score_mean | 论文展示用综合安全/舒适/效率分数 | 越高越好，但不能牺牲成功率 |
| merge_success_rate | 是否完成合流/变道目标 | 越高越好 |
| endpoint_success_rate | 终点是否成功 | 越高越好 |
| safety_success_rate | 整条轨迹是否安全 | 越高越好 |
| collision_rate | 一条轨迹碰撞一次即记为碰撞 | 越低越好 |
| min_ttc | 最小 TTC | 越高越安全 |
| min_thw | 最小车头时距 THW | 越高越安全 |
| mean_abs_jerk | 平均绝对 jerk | 越低越平顺 |

当前选模型的优先级：

```text
1. collision_rate 不能太高
2. merge / endpoint / safety 三个成功率要均衡
3. eval_ep_rew_mean 和 dense100 尽量高
4. TTC / THW 不能明显恶化
5. 不盲目使用 300 轮终点，优先保存 best checkpoint
```

## 6. 失败或不推荐路线

## 8. 后续建议

下一步不建议再大范围随机扫。更合理的是围绕当前首选模型做小范围验证：

```text
基础：U220_L5e6_D230_Decay250
候选：best_epoch_265 / best_epoch_270
重点：用同一评估脚本、同一 N=100 或更高 episode 数量复评
目标：确认 collision_rate、merge_success_rate、safety_success_rate 是否稳定
```

如果继续训练调参，优先方向：

1. 固定 `U220_L5e6_D230_Decay250` 底座，只做 seed 复验。
2. 在 250 decay 附近做极小搜索，例如 `decay_epoch=248/250/252`，不要再大改结构。
3. 尝试保存并比较 260/265/270/275，而不是只看 300。
4. 论文结果建议报告 best checkpoint 的轮次，不要默认写“最终 300 轮”。
